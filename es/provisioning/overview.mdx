---
title: "Resumen del aprovisionamiento"
description: "Cómo entran y salen los usuarios de Mintie cuando el SSO está habilitado"
---

Hay tres formas en que un usuario puede acabar con una cuenta de Mintie una vez habilitado el SSO. Elige la que coincida con las capacidades de tu IdP y tu modelo operativo.

<CardGroup cols={3}>
  <Card title="Invitación manual" icon="envelope">
    El administrador de la organización invita a los usuarios por correo electrónico. Funciona con cualquier proveedor.
    Disponible hoy.
  </Card>
  <Card title="SCIM 2.0" icon="arrows-rotate" href="/provisioning/scim">
    Tu IdP envía los eventos del ciclo de vida del usuario a Mintie. El aprovisionamiento y
    la desaprovisión se producen sin intervención del administrador. **Próximamente.**
  </Card>
  <Card title="Just-in-time" icon="bolt" href="/provisioning/jit">
    Los usuarios se crean en el primer inicio de sesión SSO, con atributos mapeados desde
    los claims SAML. **Planificado.**
  </Card>
</CardGroup>

## Comparación

| | Invitación manual | SCIM | JIT |
| --- | --- | --- | --- |
| Aprovisionamiento al contratar | Manual | Automático | En el primer inicio de sesión |
| Desaprovisionamiento al desincorporar | Manual | Automático | Manual o vía bloqueo del IdP |
| Sincronización de grupo / rol | No | Sí | Solo desde claims SAML |
| Esfuerzo de configuración | Ninguno | Medio | Bajo |
| Mejor para | Equipos pequeños | Empresas con IdP capaces de SCIM | Equipos medianos sin herramientas SCIM |

## La desaprovisión importa

Para las organizaciones que requieren SSO, la desaprovisión es un control de seguridad, no una característica de conveniencia. Un usuario desactivado con claves de API activas todavía puede llegar a la API de Mintie hasta que esas claves se revoquen: consulta [Claves de API vinculadas a la identidad de IdP](/api-and-keys/api-keys) para ver cómo se cierra esto.
