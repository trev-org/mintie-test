title: "Habilitar SSO"
description: "Conecta tu proveedor de identidad, prueba la conexión y exige SSO para tu organización"
---

La página de configuración de SSO permite a los administradores de organización gestionar su proveedor de identidad de extremo a extremo, sin necesidad de abrir un ticket con soporte de Mintie. Puedes conectar un proveedor, comprobar su salud, ejecutar un inicio de sesión de prueba real y aplicar el SSO en toda la organización.

## Abrir la página de configuración de SSO

Inicia sesión como administrador de organización y ve a **Configuración → Autenticación → SSO** (`/settings/auth/sso`).

La página tiene dos estados:

- **Estado vacío** — aún no hay un IdP conectado. Verás un selector de proveedor.
- **Estado conectado** — hay un IdP conectado. Verás su salud, un botón de probar inicio de sesión y una acción de desconectar.

El toggle **Requerir SSO** aparece debajo de ambos estados. Permanece deshabilitado hasta que tengas una conexión activa.

## Conectar un proveedor de identidad

Desde el estado vacío, elige tu proveedor:

<CardGroup cols={2}>
  <Card title="Okta" icon="plug" href="/es/setup/okta">
    Pega una URL de metadatos o sube el XML de metadatos de tu app SAML de Okta.
  </Card>
  <Card title="Google Workspace" icon="google" href="/es/setup/google-workspace">
    Introduce tu dominio principal y aprueba los scopes de SSO y Directory Sync
    como super-administrador de Google.
  </Card>
  <Card title="SAML genérico" icon="file-import" href="/es/setup/generic-saml">
    Cualquier IdP SAML 2.0. Proporciona una URL de metadatos o pega el XML.
  </Card>
  <Card title="OIDC genérico" icon="key" href="/es/setup/generic-saml">
    Cualquier IdP OIDC. Proporciona la URL del issuer, el client ID y el client secret.
  </Card>
</CardGroup>

Cada proveedor abre un formulario guiado. Al enviarlo, Mintie crea la conexión a través de WorkOS y registra un evento `sso.link` en el [registro de auditoría](/es/security/audit-logs).

### Campos requeridos por proveedor

| Proveedor | Campos requeridos |
| --- | --- |
| Okta | URL de metadatos **o** XML de metadatos |
| Google Workspace | Dominio principal (por ejemplo, `acme.com`) |
| SAML genérico | URL de metadatos **o** XML de metadatos (exactamente uno) |
| OIDC genérico | URL del issuer, client ID, client secret |

<Note>
  Para proveedores SAML, proporciona exactamente uno de `metadataUrl` o
  `metadataXml`. El formulario rechaza las peticiones que incluyan ambos o ninguno.
</Note>

## Comprobar la salud de la conexión

Una vez conectado, la página muestra:

- **Insignia de estado** — `Active`, `Awaiting verification` o `Disconnected`.
- **ID de conexión** — el identificador de conexión de WorkOS, útil al contactar soporte.
- **Última prueba** — marca de tiempo y resultado de tu inicio de sesión de prueba más reciente.

Una conexión nueva empieza como `Awaiting verification` y pasa a `Active` tras un inicio de sesión de prueba exitoso.

## Ejecutar un inicio de sesión de prueba

Haz clic en **Test sign-in** en el estado conectado. Mintie:

1. Genera una URL de autorización contra tu IdP.
2. Abre esa URL en una ventana emergente.
3. Espera el viaje de ida y vuelta del IdP y publica el resultado en la página.

La página muestra entonces ✓ éxito o ✗ fallo. Un fallo suele significar que la conexión está mal configurada: revisa los metadatos, los claims o las credenciales del cliente y vuelve a intentarlo.

<Tip>
  Si la ventana emergente está bloqueada o se cierra antes de terminar el
  viaje, la prueba reporta un fallo. Permite las ventanas emergentes para tu
  host de Mintie y vuelve a ejecutar la prueba.
</Tip>

## Requerir SSO para tu organización

La casilla **Requerir SSO** aplica el SSO a todos los miembros. Activarla abre un modal de confirmación que explica las consecuencias antes de aplicar cualquier cambio.

Cuando confirmas, Mintie:

1. Establece `sso_required = true` en tu organización.
2. Lanza el [trabajo de migración de sesiones](/es/concepts/sessions): cada sesión de contraseña activa se marca para re-autenticación a un ritmo continuo del 10% por minuto. Los miembros son redirigidos a tu IdP gradualmente durante aproximadamente los siguientes 10 minutos.
3. Bloquea nuevos inicios de sesión con contraseña. Los tokens de acceso personal existentes siguen funcionando; los nuevos tokens requieren una sesión SSO.

Desactivar el toggle restaura los inicios de sesión con contraseña. Las sesiones SSO existentes no se ven afectadas.

<Warning>
  El toggle de Requerir SSO está deshabilitado hasta que se conecte un IdP.
  Conecta un proveedor y ejecuta un inicio de sesión de prueba exitoso antes
  de aplicar SSO.
</Warning>

## Desconectar un proveedor

Haz clic en **Disconnect** en el estado conectado. Tras confirmar:

- La conexión de WorkOS se elimina.
- Se escribe un evento `sso.unlink` en el [registro de auditoría](/es/security/audit-logs).
- La página vuelve al estado vacío.

Los miembros ya autenticados permanecen autenticados hasta que expire su sesión. Los nuevos inicios de sesión SSO se rechazan inmediatamente.

## Referencia de la API de administración

La página de configuración llama a un pequeño conjunto de endpoints de administración. Puedes invocarlos directamente desde automatización servidor-a-servidor si necesitas crear scripts. Todos los endpoints requieren una sesión de administrador para la organización que llama y responden con `401 { "error": "no_org" }` cuando no hay contexto de organización.

| Método | Ruta | Propósito |
| --- | --- | --- |
| `GET` | `/api/auth/admin/sso/status` | Lee el estado actual de la conexión, el proveedor y el flag `ssoRequired`. |
| `POST` | `/api/auth/admin/sso/okta` | Conecta Okta con `metadataUrl` o `metadataXml`. |
| `POST` | `/api/auth/admin/sso/google` | Conecta Google Workspace con `{ "domain": "acme.com" }`. |
| `POST` | `/api/auth/admin/sso/saml` | Conecta un IdP SAML genérico con `metadataUrl` o `metadataXml`. |
| `POST` | `/api/auth/admin/sso/oidc` | Conecta un IdP OIDC genérico con `issuer`, `clientId`, `clientSecret`. |
| `POST` | `/api/auth/admin/sso/test` | Genera una URL de autorización de prueba. Devuelve `{ "testUrl": "..." }`. |
| `POST` | `/api/auth/admin/sso/require` | Activa o desactiva SSO en toda la organización con un booleano `required`. |
| `DELETE` | `/api/auth/admin/sso` | Desconecta el IdP activo. |

### Ejemplo: conectar un proveedor OIDC genérico

```bash
curl -X POST https://app.mintie.example/api/auth/admin/sso/oidc \
  -H "content-type: application/json" \
  --cookie "$ADMIN_COOKIE" \
  -d '{
    "issuer": "https://idp.example.com",
    "clientId": "mintie-prod",
    "clientSecret": "..."
  }'
```

### Ejemplo: leer el estado de la conexión

```bash
curl https://app.mintie.example/api/auth/admin/sso/status \
  --cookie "$ADMIN_COOKIE"
```

```json
{
  "provider": "okta",
  "connectionId": "conn_01H...",
  "status": "active",
  "ssoRequired": true,
  "lastTestedAt": "2026-04-30T18:22:11.000Z",
  "lastTestResult": "success"
}
```
