We use **HashiCorp Vault** as the source of truth for all secrets. Services consume secrets via:

- ICP-injected env vars (most common)
- Vault Agent sidecar (when files are required)
- Direct Vault API (last resort)

## Path conventions

```
secret/
├── shared/                         # Cross-team shared secrets
│   ├── stripe/
│   └── twilio/
├── prod/<service>/                 # Per-service prod secrets
│   ├── db
│   └── api-keys
├── stg/<service>/
├── dev/<service>/
└── platform/                       # Platform team
```

Service secrets are accessible only by the service's IAM role. Cross-team access requires explicit policy.

## Reading secrets in code

Don't read directly. Use `service.yaml` to inject:

```yaml
env:
  - name: STRIPE_API_KEY
    valueFrom:
      vault: secret/prod/billing-service/stripe-api-key
  - name: DB_PASSWORD
    valueFrom:
      vault: secret/prod/billing-service/db
      key: password
```

ICP rotates the env var if the underlying Vault secret changes (graceful pod restart on next deploy or via SIGHUP for hot-reload services).

## Local development

```bash
intuit secret get dev/billing-service/db --output .env
```

Local dev secrets are scoped to dev environments. They have no production privileges.

## Rotation

- **DB passwords**: rotated every 90 days automatically (Vault dynamic creds)
- **AWS keys**: short-lived STS, no static keys for service-to-service
- **API keys to 3rd parties**: rotated 180 days, manual but tracked
- **TLS certs**: 24h via Vault PKI

## What never goes in Vault

- Customer PII (use the data layer)
- Logs of any kind (Splunk)
- Source code (GitHub)
- Documentation (this KB)

## What never goes in Git

- Plaintext secrets, ever — [git-secrets](https://github.com/awslabs/git-secrets) hook is enforced pre-commit
- `.env` files with real values (templates only, named `.env.example`)
- Cert files, kubeconfigs, AWS credential files

If you accidentally commit a secret:

1. **Rotate immediately** — the secret is compromised the moment it hits Git, even if you force-push
2. Notify `it-sec@intuit.example`
3. Use `intuit git purge-secret` to scrub the history

## Audit

Every Vault access is audited and shipped to Splunk + the Security Lake. Usage spikes generate alerts.
