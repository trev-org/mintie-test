<Info>
  **Planned.** Step-up MFA is on the roadmap. Today, MFA is whatever your
  IdP enforces at sign-in time — there's no second-factor prompt inside Mintie.
</Info>

## What "step-up MFA" means here

Your IdP almost certainly enforces MFA at sign-in. That covers the front door. Step-up MFA covers the bedroom — sensitive actions inside Mintie that we want to gate even if the user's session is already valid.

## Actions likely to require step-up

The exact list will be finalized with input from product, but the candidates are:

- Generating a new API key
- Modifying billing or payment information
- Removing a member from the org
- Rotating a SCIM token
- Changing the org's SSO configuration

## Open design questions

<AccordionGroup>
  <Accordion title="WorkOS step-up vs. our own TOTP">
    WorkOS exposes a step-up flow that bounces back to the IdP. That gets
    free MFA for IdP-MFA-enabled orgs. The downside: it doesn't help orgs
    whose IdP doesn't enforce MFA. We may need both paths.
  </Accordion>
  <Accordion title="Which actions count as sensitive">
    The candidate list above is a starting point. Final scope is gated on
    PM input.
  </Accordion>
  <Accordion title="Per-org policy vs. global">
    Some orgs want every key creation gated. Others would find that
    excessive. Likely lands as an org-level policy with sane defaults.
  </Accordion>
</AccordionGroup>
