The audit log dashboard exposes the auth-event stream that Mintie already
writes server-side. Org admins can see who signed in, when, from where, and
via which IdP &mdash; plus SCIM provisioning and API-key lifecycle events.
Results are filterable, exportable, and retained long enough to satisfy
enterprise security reviews.

## Who can see it

The dashboard is visible only to users with the `admin` role on the org. It is
hidden entirely for non-admins &mdash; no greyed-out entry, no escalation
prompt. Mintie support sees the same view through the existing impersonation
flow.

## Where to find it

Navigate to **Settings → Auth → Audit logs** (`/settings/auth/audit-log`).
The default view shows the last 7 days in reverse-chronological order.

## What gets logged

The dashboard surfaces the following event kinds:

| Event | When emitted |
| --- | --- |
| `sso.signin` | User signs in via SSO (success and failure) |
| `sso.link` | Mintie user linked to an IdP identity |
| `sso.unlink` | Link removed |
| `scim.user.provisioned` | SCIM `POST /Users` succeeded |
| `scim.user.deactivated` | SCIM `DELETE /Users/{id}` succeeded |
| `apikey.issued` | API key issued |
| `apikey.revoked` | API key revoked (manually or by IdP deactivation) |

API key *usage* events are intentionally out of scope for v1 &mdash; they are
100&ndash;1000&times; higher volume and belong on a separate surface.

## Columns

Each row shows:

- **Timestamp**
- **Actor** (user or system)
- **Event kind**
- **IP / user-agent** (where applicable)
- **IdP**
- **Outcome** (success or failure, for event kinds that have an outcome)
- **Target** (the key or user the event acted on)

Click any row to open a detail panel with the full `meta` payload
pretty-printed.

## Filters

| Filter | Notes |
| --- | --- |
| Time range | Presets for 24h / 7d / 30d, plus a custom range up to your retention limit |
| Actor | User picker, multi-select |
| Event kind | Multi-select, grouped by SSO / SCIM / API keys |
| Outcome | Success or failure &mdash; applies only to event kinds with an outcome |
| IP address | Free-text substring filter |

## Export

You can export the current filtered view in two ways:

**CSV** &mdash; the **Export CSV** button respects the active filter. For
exports larger than 10,000 rows, Mintie runs the job asynchronously and emails
the requesting admin a signed download link valid for one hour.

**JSON via API** &mdash; `GET /api/audit/export` accepts the same filters as
query parameters and uses the same auth as the dashboard.

Webhook delivery to a SIEM is not in v1. Customers with that requirement
should poll the export endpoint &mdash; that pattern matches what most SecOps
teams are already doing with their other tools.

## Retention

Retention is set per plan tier and is not configurable per-org. Events are
hard-deleted on cron after the retention window.

| Plan | Retention | Export window |
| --- | --- | --- |
| Free / Starter | 30 days | 30 days |
| Business | 90 days | 90 days |
| Enterprise | 365 days | 365 days |

Customers who need longer retention should pull regularly via the export
endpoint and archive on their side.

## Performance and privacy

- p95 query latency is under 500ms for filtered queries within the last 30
  days, at 1M events per org.
- p95 export job latency is under 60s for the 90-day window.
- The read path runs off a separate read replica so it cannot block the auth
  write path.
- No PII beyond what's already in the events. No raw cookies, no tokens, no
  key material.

## What's not in v1

- Anomaly detection, alerting, or any SIEM-style correlation.
- Real-time webhook delivery of events.
- Non-auth events (settings, billing, page edits) &mdash; those belong to a
  separate, broader product-audit project.
- Cross-org views for partner / MSP scenarios.
