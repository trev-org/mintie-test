  prospects asked for this during the SSO procurement conversation.
</Info>

## What will be logged

The dashboard will surface a filterable feed of identity events for the org:

| Event | When emitted |
| --- | --- |
| `auth.signin.success` | User signs in (password or SSO) |
| `auth.signin.failure` | Sign-in fails, with redacted reason |
| `auth.sso.linked` | Mintie user linked to an IdP identity |
| `auth.sso.unlinked` | Link removed (admin action) |
| `auth.session.forced_relogin` | Session-migration job stamped a session |
| `scim.user.provisioned` | SCIM `POST /Users` succeeded |
| `scim.user.deactivated` | SCIM `DELETE /Users/{id}` succeeded |
| `apikey.created` | API key issued |
| `apikey.revoked` | API key revoked (manually or by IdP deactivation) |

## UI

The dashboard will live at **Settings → Auth → Audit logs**, filterable by:

- User
- Time range
- Event type
- Outcome (success / failure)

Rows expand to show full event metadata (IP, user agent, IdP claims, etc.).

## Export

CSV export and a webhook destination for SIEM forwarding are both on the wishlist for this milestone.
