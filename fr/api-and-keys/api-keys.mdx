<Warning>
  **Bientôt disponible.** Aujourd'hui, les clés d'API survivent à leur utilisateur
  propriétaire : si un utilisateur est désactivé dans votre IdP, ses clés
  continuent de fonctionner jusqu'à révocation manuelle. Nous comblons cette
  lacune ; le comportement décrit ci-dessous est l'état cible.
</Warning>

## Le problème que nous résolvons

Pour les organisations qui exigent le SSO, le déprovisioning d'un utilisateur dans l'IdP devrait également couper son accès à l'API Mintie. Aujourd'hui ce n'est pas le cas : les clés sont indépendantes de l'état de session.

## Ce qui change

Chaque clé d'API portera l'`idp_user_id` de l'utilisateur émetteur. À chaque appel d'API, le middleware d'authentification vérifie le statut "actif" dans l'IdP pour cet utilisateur.

| Vérification | Fréquence | Comportement en cas d'échec |
| --- | --- | --- |
| Flag de révocation local | Chaque appel | `401 invalid_key` |
| Statut actif dans l'IdP | Mis en cache, TTL de 5 minutes | `401 user_deactivated` |
| Exigence SSO de l'organisation | Chaque appel | `401 sso_required` (si l'utilisateur n'a qu'un mot de passe) |

Cela signifie qu'un utilisateur déprovisionné dans votre IdP verra ses clés renvoyer 401 dans les 5 minutes, indépendamment du moment où les clés ont été émises.

## Ce qui reste identique

  </Card>

## Migration

Lorsque le changement sera déployé, les clés existantes seront marquées avec l'`idp_user_id` de leur utilisateur propriétaire de manière paresseuse à la première utilisation. Les clés dont le propriétaire n'a pas d'`idp_user_id` (c'est-à-dire encore par mot de passe uniquement) continueront de fonctionner jusqu'à la date limite de retrait des mots de passe de l'organisation.
