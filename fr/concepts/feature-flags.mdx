---
title: "Feature flags"
description: "Comment Mintie pilote le déploiement du SSO par organisation via des feature flags, pour échelonner l'activation, tester et revenir sans redéploiement."
---

Tous les chemins de code SSO sont derrière un seul flag LaunchDarkly : `sso_enabled`. Le flag est évalué par organisation, pas par utilisateur.

## Le flag

| | |
| --- | --- |
| **Clé** | `sso_enabled` |
| **Type** | Booléen |
| **Défaut** | `false` |
| **Targeting** | Par organisation, via `orgId` |

Quand `sso_enabled = false` :

- Le bouton "Continuer avec le SSO" est masqué sur `/login`
- `/api/auth/idp-lookup` renvoie `{ ssoEnabled: false }`
- `/api/auth/sso/callback` renvoie `404`
- La page de paramètres SSO de l'interface d'administration est masquée

## Comment c'est évalué

L'évaluation côté serveur passe par `FeatureContext.forOrg(orgId)` :

```ts
const ctx = FeatureContext.forOrg(orgId);
if (await ctx.bool("sso_enabled")) {
  // SSO code path
}
```

L'évaluation côté frontend utilise le hook standard :

```tsx
const ssoEnabled = useFlag("sso_enabled");
```

## Stratégie de déploiement

Nous activons `sso_enabled` par organisation au fur et à mesure que nous intégrons les pilotes :

1. Organisation interne Mintie (déjà activée)
2. Organisations design partner (pilotes actifs)
3. Les trois deals d'entreprise bloqués (en attente d'Okta/SCIM)
4. Disponibilité générale : le flag passe à `true` par défaut

Une fois la GA atteinte, le flag restera pendant un cycle de release comme interrupteur d'urgence, puis sera supprimé.
