---
title: "Comment fonctionne le SSO"
description: "Flux de bout en bout d'une connexion SSO dans Mintie"
---

Le flux SSO de Mintie est basé sur OIDC par défaut. Les fournisseurs SAML sont tunnellisés via WorkOS, de sorte que le code de l'application ne voit jamais qu'un callback de forme OIDC.

## Le happy path

<Steps>
  <Step title="L'utilisateur saisit son e-mail sur /login">
    Le frontend appelle `/api/auth/idp-lookup` avec le domaine de l'e-mail. Si
    l'organisation a le SSO activé, la réponse inclut une URL de redirection vers l'IdP.
  </Step>
  <Step title="L'utilisateur s'authentifie auprès de l'IdP">
    Mintie ne voit jamais le mot de passe. L'IdP redirige vers
    `/api/auth/sso/callback` avec un code d'autorisation.
  </Step>
  <Step title="Mintie valide le callback">
    Le handler du callback valide le jeton `state` (CSRF), le `nonce` (rejeu),
    ainsi que la signature, l'émetteur, l'audience et l'expiration de l'ID token.
  </Step>
  <Step title="Mintie lie ou crée l'utilisateur">
    La recherche de l'utilisateur se fait d'abord par `(idp_provider, idp_user_id)`, puis par e-mail.
    Voir [liaison d'identité](/fr/concepts/identity-linking) pour les règles.
  </Step>
  <Step title="Mintie émet une session">
    Un cookie de session est défini avec `HttpOnly`, `Secure` et `SameSite=Lax`.
  </Step>
</Steps>

## Garanties de sécurité

- Les ID tokens sont vérifiés cryptographiquement par rapport au JWKS de l'IdP
- Les jetons `state` sont à usage unique et liés à la session d'origine
- Les cookies de session ne sont pas lisibles depuis JavaScript
- Les identifiants de l'IdP n'atteignent jamais les serveurs de Mintie
