<Info>
  **Planifié.** Le MFA renforcé est sur la feuille de route. Aujourd'hui, le MFA est ce que votre
  IdP impose au moment de la connexion — il n'y a pas de prompt de second facteur dans Mintie.
</Info>

## Ce que "MFA renforcé" signifie ici

Votre IdP impose presque certainement le MFA à la connexion. Cela couvre la porte d'entrée. Le MFA renforcé couvre la chambre — les actions sensibles dans Mintie que nous voulons protéger même si la session de l'utilisateur est déjà valide.

## Actions susceptibles d'exiger un renforcement

La liste exacte sera finalisée avec l'apport du produit, mais les candidats sont :

- Générer une nouvelle clé d'API
- Modifier les informations de facturation ou de paiement
- Retirer un membre de l'organisation
- Faire tourner un jeton SCIM
- Modifier la configuration SSO de l'organisation

## Questions de conception ouvertes

<AccordionGroup>
  <Accordion title="Step-up WorkOS vs notre propre TOTP">
    WorkOS expose un flux step-up qui rebondit vers l'IdP. Cela donne
    du MFA gratuit pour les organisations dont l'IdP a le MFA activé. L'inconvénient : cela n'aide pas les organisations
    dont l'IdP n'impose pas le MFA. Il faudra peut-être les deux chemins.
  </Accordion>
  <Accordion title="Quelles actions comptent comme sensibles">
    La liste de candidats ci-dessus est un point de départ. Le périmètre final dépend de
    l'apport du PM.
  </Accordion>
  <Accordion title="Politique par organisation vs globale">
    Certaines organisations veulent que chaque création de clé soit protégée. D'autres trouveraient cela
    excessif. Probablement une politique au niveau de l'organisation avec des valeurs par défaut sensées.
  </Accordion>
</AccordionGroup>
