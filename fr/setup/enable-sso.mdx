La page de paramètres SSO permet aux administrateurs d'organisation de gérer leur fournisseur d'identité de bout en bout, sans ouvrir de ticket auprès du support Mintie. Vous pouvez connecter un fournisseur, vérifier sa santé, exécuter une vraie connexion de test et imposer le SSO à toute l'organisation.

## Ouvrir la page de paramètres SSO

Connectez-vous en tant qu'administrateur d'organisation et accédez à **Paramètres → Authentification → SSO** (`/settings/auth/sso`).

La page a deux états :

- **État vide** — aucun IdP n'est connecté. Vous voyez un sélecteur de fournisseur.
- **État connecté** — un IdP est connecté. Vous voyez sa santé, un bouton de test de connexion et une action de déconnexion.

Le toggle **Exiger le SSO** se trouve sous les deux états. Il reste désactivé tant que vous n'avez pas de connexion active.

## Connecter un fournisseur d'identité

Depuis l'état vide, choisissez votre fournisseur :

  <Card title="Okta" icon="plug" href="/fr/setup/okta">
    Collez une URL de métadonnées ou téléversez le XML de métadonnées de votre app SAML Okta.
  </Card>
  <Card title="Google Workspace" icon="google" href="/fr/setup/google-workspace">
    Saisissez votre domaine principal et approuvez les scopes SSO et Directory Sync
    en tant que super-administrateur Google.
  </Card>
  <Card title="SAML générique" icon="file-import" href="/fr/setup/generic-saml">
    Tout IdP SAML 2.0. Fournissez une URL de métadonnées ou collez le XML.
  </Card>
  <Card title="OIDC générique" icon="key" href="/fr/setup/generic-saml">
    Tout IdP OIDC. Fournissez l'URL de l'issuer, le client ID et le client secret.
  </Card>
</CardGroup>

Chaque fournisseur ouvre un formulaire guidé. Lors de la soumission, Mintie crée la connexion via WorkOS et écrit un événement `sso.link` dans le [journal d'audit](/fr/security/audit-logs).

### Champs requis par fournisseur

| Fournisseur | Champs requis |
| --- | --- |
| Okta | URL de métadonnées **ou** XML de métadonnées |
| Google Workspace | Domaine principal (par exemple, `acme.com`) |
| SAML générique | URL de métadonnées **ou** XML de métadonnées (exactement un) |
| OIDC générique | URL de l'issuer, client ID, client secret |

<Note>
  Pour les fournisseurs SAML, fournissez exactement un de `metadataUrl` ou
  `metadataXml`. Le formulaire rejette les requêtes qui incluent les deux ou
  aucun.
</Note>

## Vérifier la santé de la connexion

Une fois connecté, la page affiche :

- **Badge de statut** — `Active`, `Awaiting verification` ou `Disconnected`.
- **ID de connexion** — l'identifiant de connexion WorkOS, utile lors du contact avec le support.
- **Dernier test** — horodatage et résultat de votre connexion de test la plus récente.

Une nouvelle connexion commence en `Awaiting verification` et bascule en `Active` après une connexion de test réussie.

## Exécuter une connexion de test

Cliquez sur **Test sign-in** dans l'état connecté. Mintie :

1. Génère une URL d'autorisation vers votre IdP.
2. Ouvre cette URL dans une fenêtre popup.
3. Attend l'aller-retour de l'IdP et publie le résultat sur la page.

La page affiche ensuite ✓ succès ou ✗ échec. Un échec signifie généralement que la connexion est mal configurée — revérifiez les métadonnées, les claims ou les identifiants client, puis réessayez.

<Tip>
  Si le popup est bloqué ou fermé avant la fin de l'aller-retour, le test
  signale un échec. Autorisez les popups pour votre hôte Mintie et relancez le
  test.
</Tip>

## Exiger le SSO pour votre organisation

La case **Exiger le SSO** impose le SSO à tous les membres. L'activer ouvre une modale de confirmation qui explique les conséquences avant tout changement.

Lorsque vous confirmez, Mintie :

1. Définit `sso_required = true` sur votre organisation.
2. Déclenche le [job de migration de sessions](/fr/concepts/sessions) — chaque session de mot de passe active est marquée pour réauthentification à un rythme continu de 10 % par minute. Les membres sont redirigés vers votre IdP progressivement sur environ les 10 prochaines minutes.
3. Bloque les nouvelles connexions par mot de passe. Les jetons d'accès personnels existants continuent de fonctionner ; les nouveaux jetons nécessitent une session SSO.

Désactiver le toggle restaure les connexions par mot de passe. Les sessions SSO existantes ne sont pas affectées.

<Warning>
  Le toggle Exiger le SSO est désactivé tant qu'aucun IdP n'est connecté.
  Connectez un fournisseur et exécutez une connexion de test réussie avant
  d'imposer le SSO.
</Warning>

## Déconnecter un fournisseur

Cliquez sur **Disconnect** dans l'état connecté. Après confirmation :

- La connexion WorkOS est démantelée.
- Un événement `sso.unlink` est écrit dans le [journal d'audit](/fr/security/audit-logs).
- La page revient à l'état vide.

Les membres déjà connectés restent connectés jusqu'à l'expiration de leur session. Les nouvelles connexions SSO sont rejetées immédiatement.

## Référence de l'API d'administration

La page de paramètres appelle un petit ensemble d'endpoints d'administration. Vous pouvez les appeler directement depuis de l'automatisation serveur-à-serveur si vous devez scripter le provisioning. Tous les endpoints nécessitent une session admin pour l'organisation appelante et répondent avec `401 { "error": "no_org" }` lorsqu'aucun contexte d'organisation n'est présent.

| Méthode | Chemin | Objet |
| --- | --- | --- |
| `GET` | `/api/auth/admin/sso/status` | Lit le statut actuel de la connexion, le fournisseur et le flag `ssoRequired`. |
| `POST` | `/api/auth/admin/sso/okta` | Connecte Okta avec `metadataUrl` ou `metadataXml`. |
| `POST` | `/api/auth/admin/sso/google` | Connecte Google Workspace avec `{ "domain": "acme.com" }`. |
| `POST` | `/api/auth/admin/sso/saml` | Connecte un IdP SAML générique avec `metadataUrl` ou `metadataXml`. |
| `POST` | `/api/auth/admin/sso/oidc` | Connecte un IdP OIDC générique avec `issuer`, `clientId`, `clientSecret`. |
| `POST` | `/api/auth/admin/sso/test` | Génère une URL d'autorisation de test. Renvoie `{ "testUrl": "..." }`. |
| `POST` | `/api/auth/admin/sso/require` | Active ou désactive le SSO à l'échelle de l'organisation avec un booléen `required`.|
| `DELETE` | `/api/auth/admin/sso` | Déconnecte l'IdP actif. |

### Exemple : connecter un fournisseur OIDC générique

```bash
curl -X POST https://app.mintie.example/api/auth/admin/sso/oidc \
  -H "content-type: application/json" \
  --cookie "$ADMIN_COOKIE" \
  -d '{
    "issuer": "https://idp.example.com",
    "clientId": "mintie-prod",
    "clientSecret": "..."
  }'
```

### Exemple : lire l'état de la connexion

```bash
curl https://app.mintie.example/api/auth/admin/sso/status \
  --cookie "$ADMIN_COOKIE"
```

```json
{
  "provider": "okta",
  "connectionId": "conn_01H...",
  "status": "active",
  "ssoRequired": true,
  "lastTestedAt": "2026-04-30T18:22:11.000Z",
  "lastTestResult": "success"
}
```
