---
title: "SAML / OIDC générique"
description: "Connectez n'importe quel fournisseur d'identité SAML 2.0 ou OIDC à Mintie via une configuration générique, avec métadonnées, certificats et mappage d'attributs."
---

Si votre IdP n'est pas Okta ou Google Workspace, le connecteur SAML / OIDC générique le couvre. Mintie tunnellise les deux protocoles via WorkOS, de sorte que le code de l'application ne voit jamais qu'un callback de forme OIDC.

<Note>
  La configuration aujourd'hui est faite par votre équipe de compte Mintie. Une fois la
  [interface d'administration](/fr/setup/enable-sso) publiée, vous le ferez vous-même.
</Note>

## Connexion OIDC

Vous aurez besoin de :

- Émetteur (`iss`)
- URL d'autorisation
- URL de jeton
- URL JWKS
- ID client et secret client
- Scopes requis — Mintie a besoin au minimum de `openid email profile`

## Connexion SAML 2.0

Vous aurez besoin de :

- XML de métadonnées du fournisseur d'identité (ou l'URL de métadonnées)
- Le certificat de signature de l'IdP (intégré aux métadonnées)
- Claims d'attributs requis :
  - `email` (requis)
  - `firstName` et `lastName` (optionnels mais recommandés)
  - `groups` (optionnel, utilisé pour le mappage de rôles)

## Endpoints Mintie

Vous pointerez votre IdP vers :

| Endpoint Mintie | URL |
| --- | --- |
| ACS (SAML) | `https://app.mintie.example/api/auth/saml/acs` |
| URI de redirection (OIDC) | `https://app.mintie.example/api/auth/sso/callback` |
| ID d'entité | `https://app.mintie.example/auth/<your-org-slug>` |

## Validation

Mintie valide la signature, l'émetteur, l'audience et l'expiration de chaque callback. Les IdP mal configurés sont signalés avec des codes d'erreur spécifiques, pas un générique "échec de connexion" — voir les [journaux d'audit](/fr/security/audit-logs) une fois publiés.
