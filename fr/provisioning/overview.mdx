---
title: "Vue d'ensemble du provisioning"
description: "Comment les utilisateurs entrent et sortent de Mintie quand le SSO est activé"
---

Il existe trois façons pour un utilisateur de se retrouver avec un compte Mintie une fois le SSO activé. Choisissez celle qui correspond aux capacités de votre IdP et à votre modèle opérationnel.

<CardGroup cols={3}>
  <Card title="Invitation manuelle" icon="envelope">
    L'administrateur de l'organisation invite les utilisateurs par e-mail. Fonctionne avec n'importe quel fournisseur.
    Disponible aujourd'hui.
  </Card>
  <Card title="SCIM 2.0" icon="arrows-rotate" href="/provisioning/scim">
    Votre IdP envoie les événements de cycle de vie utilisateur à Mintie. Le provisioning et
    le déprovisioning se font sans intervention administrative. **Bientôt disponible.**
  </Card>
  <Card title="Juste-à-temps" icon="bolt" href="/provisioning/jit">
    Les utilisateurs sont créés à la première connexion SSO, avec des attributs mappés depuis
    les claims SAML. **Planifié.**
  </Card>
</CardGroup>

## Comparaison

| | Invitation manuelle | SCIM | JIT |
| --- | --- | --- | --- |
| Provisioning à l'embauche | Manuel | Automatique | À la première connexion |
| Déprovisioning au départ | Manuel | Automatique | Manuel ou via blocage IdP |
| Synchro groupe / rôle | Non | Oui | Depuis les claims SAML uniquement |
| Effort de configuration | Aucun | Moyen | Faible |
| Idéal pour | Petites équipes | Entreprises avec IdP capables de SCIM | Équipes de taille moyenne sans outil SCIM |

## Le déprovisioning compte

Pour les organisations qui exigent le SSO, le déprovisioning est un contrôle de sécurité, pas une fonctionnalité de confort. Un utilisateur désactivé avec des clés d'API actives peut toujours appeler l'API Mintie tant que ces clés ne sont pas révoquées — voir [Clés d'API liées à l'identité IdP](/api-and-keys/api-keys) pour savoir comment cela se ferme.
