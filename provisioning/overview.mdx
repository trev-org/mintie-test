There are three ways a user can end up with a Mintie account once SSO is enabled. Pick the one that matches your IdP capabilities and operational model.

<CardGroup cols={3}>
  <Card title="Manual invite" icon="envelope">
    The org admin invites users by email. Works with any provider.
    Available today.
  </Card>
  <Card title="SCIM 2.0" icon="arrows-rotate" href="/provisioning/scim">
    Your IdP pushes user lifecycle events to Mintie. Provision and
    deprovision happen without admin intervention. **Coming soon.**
  </Card>
  <Card title="Just-in-time" icon="bolt" href="/provisioning/jit">
    Users are created on first SSO sign-in, with attributes mapped from
    SAML claims. **Planned.**
  </Card>
</CardGroup>

## Comparison

| | Manual invite | SCIM | JIT |
| --- | --- | --- | --- |
| Provision on hire | Manual | Automatic | On first sign-in |
| Deprovision on offboard | Manual | Automatic | Manual or via IdP block |
| Group / role sync | No | Yes | From SAML claims only |
| Setup effort | None | Medium | Low |
| Best for | Small teams | Enterprises with SCIM-capable IdPs | Mid-size teams without SCIM tooling |

## Deprovisioning matters

For SSO-required orgs, deprovisioning is a security control, not a convenience feature. A deactivated user with active API keys can still hit the Mintie API until those keys are revoked — see [API keys tied to IdP identity](/api-and-keys/api-keys) for how that closes.
