The SSO settings page lets org admins manage their identity provider end-to-end — no Mintie support ticket required. You can connect a provider, check its health, run a real test sign-in, and enforce SSO across the org.

## Open the SSO settings page

Sign in as an org admin and go to **Settings → Auth → SSO** (`/settings/auth/sso`).

The page has two states:

- **Empty state** — no IdP is connected yet. You see a provider picker.
- **Connected state** — an IdP is connected. You see its health, a Test sign-in button, and a Disconnect action.

The **Require SSO** toggle sits below both states. It stays disabled until you have an active connection.

## Connect an identity provider

From the empty state, choose your provider:

  <Card title="Okta" icon="plug" href="/setup/okta">

  </Card>
  <Card title="Google Workspace" icon="google" href="/setup/google-workspace">
    Enter your primary domain and approve the SSO and Directory Sync scopes
    as a Google super-admin.
  </Card>
  <Card title="Generic SAML" icon="file-import" href="/setup/generic-saml">
    Any SAML 2.0 IdP. Provide a metadata URL or paste the metadata XML.
  </Card>
  <Card title="Generic OIDC" icon="key" href="/setup/generic-saml">
    Any OIDC IdP. Provide the issuer URL, client ID, and client secret.
  </Card>
</CardGroup>

Each provider opens a guided form. When you submit, Mintie creates the connection through WorkOS and writes an `sso.link` event to the [audit log](/security/audit-logs).

### Required inputs by provider

| Provider | Required fields |
| --- | --- |
| Okta | Metadata URL **or** metadata XML |
| Google Workspace | Primary domain (for example, `acme.com`) |
| Generic SAML | Metadata URL **or** metadata XML (exactly one) |
| Generic OIDC | Issuer URL, client ID, client secret |

<Note>
  For SAML providers, supply exactly one of `metadataUrl` or `metadataXml`. The
  form rejects requests that include both or neither.
</Note>

## Check connection health

Once connected, the page shows:

- **Status badge** — `Active`, `Awaiting verification`, or `Disconnected`.
- **Connection ID** — the WorkOS connection identifier, useful when contacting support.
- **Last test** — timestamp and result of your most recent test sign-in.

A new connection starts as `Awaiting verification` and flips to `Active` after a successful test sign-in.

## Run a test sign-in

Click **Test sign-in** in the connected state. Mintie:

1. Generates an authorize URL against your IdP.
2. Opens that URL in a popup window.
3. Waits for the IdP round-trip and posts the result back to the page.

The page then shows either ✓ success or ✗ failure. A failure usually means the connection is misconfigured — recheck the metadata, claims, or client credentials, then try again.

<Tip>
  If the popup is blocked or closed before the round-trip finishes, the test
  reports a failure. Allow popups for your Mintie host and re-run the test.
</Tip>

## Require SSO for your org

The **Require SSO** checkbox enforces SSO for every member. Enabling it opens a confirmation modal that explains the consequences before any change happens.

When you confirm, Mintie:

1. Sets `sso_required = true` on your organization.
2. Triggers the [session-migration job](/concepts/sessions) — every active password session is stamped for re-auth at a rolling 10% per minute. Members are redirected to your IdP gradually over roughly the next 10 minutes.
3. Blocks new password sign-ins. Existing personal access tokens keep working; new tokens require an SSO session.

Disabling the toggle restores password sign-ins. Existing SSO sessions are not affected.

<Warning>
  The Require SSO toggle is disabled until an IdP is connected. Connect a
  provider and run a successful test sign-in before enforcing SSO.
</Warning>

## Disconnect a provider

Click **Disconnect** in the connected state. After confirming:

- The WorkOS connection is torn down.
- An `sso.unlink` event is written to the [audit log](/security/audit-logs).
- The page returns to the empty state.

Members already signed in stay signed in until their session expires. New SSO sign-ins are rejected immediately.

## Admin API reference

The settings page calls a small set of admin endpoints. You can call them directly from server-to-server automation if you need to script provisioning. All endpoints require an admin session for the calling org and respond with `401 { "error": "no_org" }` when no org context is present.

| Method | Path | Purpose |
| --- | --- | --- |
| `GET` | `/api/auth/admin/sso/status` | Read the current connection status, provider, and `ssoRequired` flag. |
| `POST` | `/api/auth/admin/sso/okta` | Connect Okta with `metadataUrl` or `metadataXml`. |
| `POST` | `/api/auth/admin/sso/google` | Connect Google Workspace with `{ "domain": "acme.com" }`. |
| `POST` | `/api/auth/admin/sso/saml` | Connect a generic SAML IdP with `metadataUrl` or `metadataXml`. |
| `POST` | `/api/auth/admin/sso/oidc` | Connect a generic OIDC IdP with `issuer`, `clientId`, `clientSecret`. |
| `POST` | `/api/auth/admin/sso/test` | Generate a test authorize URL. Returns `{ "testUrl": "..." }`. |
| `POST` | `/api/auth/admin/sso/require` | Toggle org-wide SSO with a `required` boolean. |
| `DELETE` | `/api/auth/admin/sso` | Disconnect the active IdP. |

### Example: connect a generic OIDC provider

```bash
curl -X POST https://app.mintie.example/api/auth/admin/sso/oidc \
  -H "content-type: application/json" \
  --cookie "$ADMIN_COOKIE" \
  -d '{
    "issuer": "https://idp.example.com",
    "clientId": "mintie-prod",
    "clientSecret": "..."
  }'
```

### Example: read connection status

```bash
curl https://app.mintie.example/api/auth/admin/sso/status \
  --cookie "$ADMIN_COOKIE"
```

```json
{
  "provider": "okta",
  "connectionId": "conn_01H...",
  "status": "active",
  "ssoRequired": true,
  "lastTestedAt": "2026-04-30T18:22:11.000Z",
  "lastTestResult": "success"
}
```
